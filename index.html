  • [+] Prepare for Linear Algebra
    December 18th, 2020

    Linear Algebra studies properties, and approaches to solving, systems of linear equations, or linear systems. A linear equation can be expressed as a1x1+a2x2+...+anxn= b, where {x1, x2,…, xn} are variables, and {a1, a2,…, an} are, each, a set of coefficients corresponding to each variable. Linear equations within the same system are expressed with the same variables. A solution to the linear system is a set of values {s1, s2,…, sn} which, when substituted into the corresponding variables, make each linear equation true. A system may have more than one solution, and the set of all possible solutions is a solution set. One approach to solving a system of linear equations expresses one linear equation in terms of a single variable, then substitutes the expression into instances of the same variable within other equations of the system, repeating the process until a solution can be determined. Linear Algebra studies an approach that permits solutions for all variables within a system to be computed simultaneously, using algorithms called row reduction operations.


    The coefficients of each linear equation can be represented by a row within a matrix. Row reduction operations, which are applied to a matrix, are computationally efficient algorithms involving basic arithmetic. The elementary row reduction operations are: a. replacing one row by the sum of itself and the scale of another row; b. interchanging the order of two rows; c. scalar multiplying a row by a nonzero constant. Row reduction operations are applied in such a way as to manipulate the entries of the matrix to embody a particular form: namely, row echelon (zeros beneathe and to the left of each leading entry) and reduced row echelon (zeros beneath, to the left of and above each leading entry, where each leading entry must also be the number one) . From the properties of these forms can be inferred properties of the associated system of equations, including solutions of the system. A matrix that positions the b value, from the linear equation a1x1+a2x2+...+anxn= b, into the rightmost column of the matrix is called an augmented matrix. After reduction to reduced row echelon form, the manipulated values of the augmented matrix indicate the solution for x, or {x1, x2,…, xn}.


    Take, for example, a system of linear equations {2x1+6x2=6, x1+2x2=2}. In augmented matrix form, the two rows may be interchanged, moving [1 2 2] from the second row to the first row, and [2 6 6] from the first row to the second row. The second row of [2 6 6] may be replaced with the sum of itself, and the first row of [1 2 2] scaled by a factor of -2, to yield [0 2 2] for the second row and [1 2 2] for the first row. The form of this matrix is called echelon form, and indicates the location of the pivots, or leading entries, for each row (1st column in the first row; 2nd column in the 2nd row). The second row may then be scaled by a factor of ½ to yield [0 1 1] for the second row. Replacing the first row with the sum of itself, and the second row scaled by a factor of -2, yields [1 0 0] for the first row and [0 1 1] for the second row. The form of this matrix is called reduced row echelon form, and indicates the solution for row equivalent matrices. Row equivalent matrices are matrices that, with any combination of row reduction operations, yield the same matrix in reduced row echelon form. Here, the first row indicates that x1 equals 0 and the second row indicates that x2 equals 1. Thus, the solution of the linear system is {0, 1}.


    The linear equation a1x1+a2x2+...+anxn= b can be expressed in different forms: as a vector equation, a matrix equation, or a linear transformation. As a vector equation, a 1 is the first column, a2 is the second column, and b is the last column of the augmented matrix in reduced row echelon form i.e. [1, 0]x1 + [0, 1]x2 = [0, 1]. (In the preceding example and hereforth, a comma inside of a matrix indicates a separate row, so, for example, two matrix entries separated by a comma indicates a two row, single column matrix i.e. a column vector.) As a matrix equation Ax=b, matrix A is composed as [a1 a2 an] and the augmented column of the reduced row echelon form of [ A | b ] is x. As a linear transformation Ax=T(x), the image of x under the action of T is given by T(x), where T can be viewed as the mapping of x to b. The various forms of linear equation a1x1+a2x2+...+anxn= b yield distinct, but interconnected, insights.


    • When the reduced row echelon form of an augmented matrix with m rows and n columns contains a pivot in the rightmost column, (e.g. [1 2 2, 2 4 6] reduces to [1 2 2, 0 0 2] where 0 ≠ 2): then no value for b in Rm for the linear equation a1x1+a2x2+...+anxn= b has a solution in Rn, and the system is inconsistent.


    • When the reduced row echelon form of an augmented matrix with m rows and n columns does not contain a pivot in the rightmost column and does not contain a pivot in each row (e.g. [1 2 2, 2 4 4] reduces to [1 2 2, 0 0 0], where x1 equals 2 – 2x2): then some real values for b in Rm for the linear equation a1x1+a2x2+...+anxn= b have solutions in Rn where b is a linear combination of the vectors {a1, a2,…, an}, the system is consistent, and at least one element of the set {a1, a2,…, an} is a linear combination of the other elements of the set. In other words, an element of the set {a1, a2,…, an} is the sum of scalar multiples of other elements of the set.


    • When the reduced row echelon form of an augmented matrix with m rows and n columns does not contain a pivot in the rightmost column, does contain a pivot in each row, and, where b is the zero vector, a1x1+a2x2+...+anxn= 0 contains a nontrivial solution (e.g. [1 2 2 0, 1 2 6 0] reduces to [1 2 0 0, 0 0 1 0], with x1 equal to -2x2 and x3 equal to zero): then every value for b in Rm for the linear equation a1x1+a2x2+...+anxn= b has a solution where b is a linear combination of the vectors {a1, a2,…, an}, the system is consistent, T maps each x in Rn to at least one b in Rm, no elements of the set {a1, a2,…, an} are linear combinations of other elements of the set, and the span (the set of all linear combinations) of vectors {a1, a2,…, an} is in Rm.


    • When the reduced row echelon form of an augmented matrix with m rows and n columns does not contain a pivot in the rightmost column, does contain a pivot in each row, and, where b is the zero vector, a1x1+a2x2+...+anxn= 0 contains only the trivial solution (e.g. [1 2 0, 2 6 0] reduces to [1 0 0, 0 1 0], where x1 and x2 both equal zero): then every value for b in Rm for the linear equation a1x1+a2x2+...+anxn= b has a solution where b is a linear combination of the vectors {a1, a2,…, an}, the system is consistent, T maps each x in Rn to only one b in Rm, no elements of the set {a1, a2,…, an} are linear combinations of other elements of the set, the span of vectors {a1, a2,…, an} coincides with Rm, and the vectors {a1, a2,…, an} are linearly independent.


    The initial concepts introduced in Linear Algebra form the basis that spans all subsequent concepts. This is particularly true when vectors of the vector equation are connoted as vector spaces, when the linear combination b of the matrix equation takes the form of the eigenvector scaled by an eigenvalue, and, for linear transformations, when x becomes the kernel (or null space) which maps to the zero vector, and T(x) the range (or column space) which maps from a vector x. The study of Linear Algebra involves viewing much of the same phenomena from different perspectives where they can be applied to different types of problems. Gaining the early intuition makes subsequent learning less of a surprise and more of a natural progression. Meanwhile, the unfolding of new considerations adds depth and meaning to prior learning, so regularly surveying learned concepts may be helpful for better appreciating interconnections.

  • [+] Avoid Calculus errors
    June 10th, 2020

    Foundations


    When applying trigonometric identities, a common mistake is to forget or confuse formulas as well as the intuition behind their construction. Remember that, where intuition might fail, higher-level patterns might instead be used to reconstruct formulas and aid recall.


    In some cases, patterns extend across identities:


    Both the half-angle and power-reducing formulas involve functions f(t) = (1 – cos(t))/2 and g(t) = (1 + cos(t))/2. For both sets of identities, the formulas corresponding to: sin invoke f(t), cos invoke g(t), and tan invoke f(t)/g(t). Half-angle formulas take the square root of, while power-reducing formulas double the parameter value of, each invocation.


    The angle sum/difference formulas compose the double angle formulas, so committing the latter to memory is unnecessary.


    Patterns can take on any number of forms and scope. A direct relationship between sum/difference-to-product and product-to-sum/difference identities can be more difficult to establish. As one example:


    On either side of the equation are a product of two trigonometric functions and a binomial of two trigonometric functions.

    On the left side of the equation (i.e. initial expression), the parameter values are angles a and b.

    On the right side of the equation (i.e. final expression), the parameter values are angles (a +/– b) and (a –/+ b).

    For a product of both sine and cosine, the equivalent binomial is a form of sine +/– sine, and the angles on the right side of the equation are (a + b) and (a – b).

    For a product of either sine or cosine, the equivalent binomial is a form of cosine +/– cosine, and the angles on the right side of the equation are (a – b) and (a + b).

    For a product where the second function is cosine, the terms of the equivalent binomial are added.

    For a product where the second function is sine, the terms of the equivalent binomial are subtracted.

    For a binomial on the left side of the equation, the product on the right side of the equation is multiplied by two, and each angle is divided by two.

    For a product on the left side of the equation, each term of the (or, the entire) binomial on right side of the equation is divided by two.

    For a binomial on the left side of the equation where both functions are cosine, the product on the right side of the equation is also multiplied by negative one.


    Here, developing the basic intuition, if not a separate set of patterns for each identity, would likely be more efficient.

    Integrals


    When applying u-substitution to an expression, a common mistake is to forget or confuse elements of the dx substitute when attempting the operation mentally. Remember to write out the factorization required to convert to du, as frequently doing otherwise could raise the likelihood of oversight when a more complex factorization is entailed.


    For example:

    [3(1 – x/7)3 – 2(1 – x/7)3]x dx

    u = 1 – (x/7); du = (-1/7) dx

    [3(1 – x/7)3 – 2(1 – x/7)3]x (-7)(-1/7) dx

    [3u3 – 2u3]x (-7) du

    x = 7(1 – u)

    [3u3 – 2u3](7(1 – u))(-7) du

    When re-applying integration by parts to an expression xy, where u equals x and v equals the antiderivative of y, a common mistake is to set u equal to the y-term. Remember that, in subsequent applications of integration by parts, u equals the derivative of the x-term from the previous step.


    –


    When integrating exponential functions, where xa becomes ( xa + 1/(a + 1)), a common mistake is to omit (1/(a + 1)) after converting xa to xa + 1. In that case, it may be helpful to always write (xa + 1/(a + 1)) as (1/(a + 1))( xa + 1) where deliberately writing (1/(a + 1)) first may help prevent oversight.


    –


    When integrating trigonometric and/or exponential functions where one of the limits of integration is 0, a common mistake is to dismiss the integral evaluated at the limit of 0. Remember to evaluate each term at both limits even if the integrand does not contain trigonometric or exponential functions, as frequent dismissal, even where valid in some cases, makes mental error in other cases more likely.


    –


    When integrating to find area A by ∫ f dA or volume V by ∫ f dV where f is bound by shape S, a common mistake is to forget how to simplify the expression without formal integration. Remember that when f is a constant K that ∫ dA can be converted to A(S) and ∫ dV can be converted to V(S), such that V = K ∫ dA = K ( A(S)) and V = K ∫ dV = K ( V(S)).


    –


    When converting from Cartesian to spherical coordinate system and f is bound by shape S which is a cone, a common mistake is to determine Φ by the assumption that S is a cone with right angle at the vertex and thus Φ = (π / 4). Remember when computing limits of integration for Φ where S is defined as cone z = a√(x2 + y2) that the outer edge of the cone forms an angle Φ with the horizontal equal to tan-1(1/a), which is found intuitively by converting variables from Cartesian to spherical within the equation of the cone.


    For example:

    z = a√(x2 + y2)

    ρ cos(Φ) = a r;

    x2 + y2 = r2

    [ρ cos(θ) sin( Φ)]2 + [ρ sin(θ) sin( Φ)]2 = r2

    [cos2(θ) + sin 2(θ)][ρ sin(Φ)]2 = r2

    ρ sin( Φ) = r

    ρ cos(Φ) = a ρ sin( Φ)

    1 = a [ρ sin( Φ)] / [ρ cos(Φ)]

    (1/a) = tan(Φ)

    Φ = tan -1(1/a)


    Where the vertex angle is (π / 2): a = 1 and Φ = tan-1(1) = (π / 4).

    –


    When parameterizing a cone provided only its graphical features, a common mistake is to fail to convert all of its features into an equation. Remember that, when parameterizing in terms of radius s and angle t, that for radius s, base radius a and height h, z = h[(1/a)s], and when the cone is inverted, z = h[1 – (1/a)s]. When parameterizing in terms of x and y, s would be expressed as √(x2 + y2). Hence, r(s,t) = <scos(t), ssin(t), h[1 – (1/a)s]> and r(x,y) = <x, y, h[1 – (1/a)√(x2 + y2) ]>.

    When converting from Cartesian to either polar, cylindrical or spherical coordinate system, a common mistake is to forget to carry forward variables involved in the conversion of dA ( r for polar) or dV (r for cylindrical and ρ2sin(Φ) for spherical). Remember that the integrands f dA or f dV can be written as r [f] drdθ for polar or r [f] dzdrdθ for cylindrical and ρ2sin(Φ) [f] dΦdθ for spherical, where the brackets distinguish those elements of the expression that are due for integration.


    For example:

    [x2 + y2 + z2] dV = ρ2sin(Φ) [ρ2] dρdΦdθ = sin(Φ) [ρ4] dρdΦdθ

    Vectors


    When visually assessing the direction of curl for a vector field F, a common mistake is to misapply the right-hand rule and reverse the direction. Remember this alternate approach:


    a. For each axis, visualize (i.e. tilt, for a non-vertical axis) the graph to where the axis is situated in the vertical position (typically associated with z). For example, if the y-axis was horizontal, tilt all the elements in the graph along with the y-axis until y is vertical (assuming the typical position of z). While so doing, view the vector field against that axis from a two-dimensional perspective (agnostic to one of two horizontal axes), determining whether individual vectors in the field cross each other. For example, a cylindrical rotation about that axis would in two dimensions demonstrate crossing between individual vectors that, in three dimensions, would be in front of and behind the axis. Crossing indicates rotation.


    b. When rotation is identified, again tilt that vertical axis down so that it is perpendicular to your line of vision. For example, the axis would point directly toward your eyes on the positive side and directly away from your eyes on the negative side. View the vector field from this perspective and visualize drawing a circle around the axis, following the direction of the vectors. Assess whether that circle is being drawn in a clockwise or counterclockwise direction. That direction indicates the direction of curl (clockwise is negative; counterclockwise is positive).


    –


    When computing the surface area of surface S (alone or as part of the surface integral for a scalar function) represented parametrically in cylindrical or spherical coordinates, a common mistake is to express dA as rdrdt or ρ2sin(Φ)dΦdθ. Remember that dA only includes r or ρ2sin(Φ) if S is parameterized and the partial derivatives are expressed in Cartesian coordinates and converted to cylindrical or spherical coordinates by replacing instances of Cartesian variables with their cylindrical or spherical counterparts.


    –


    When computing the surface integral against a surface S that is expressed in naturalized form z = f(x,y), a common mistake is to apply the formula | rx x ry | = √[(dz/dx)2 + (dz/dy)2 + 1] for dS where dS is assumed to be | rx x ry | dA before recognizing that the surface integral is for a vector field. Remember that a surface integral for a vector field, where dS equals n * | rx x ry | dA = [ ( rx x ry ) / | rx x ry | ] * | rx x ry | dA = ( rx x ry ) dA, ( rx x ry ) can be computed from | rx x ry | where its application would have otherwise been appropriate by recognizing that ( rx x ry ) = < –(dz/dx), –( dz/dy), 1 >.


    –


    When computing the surface integral for a vector field against surface S, where S represents a spherical surface, ρ represents fluid density, v represents a velocity field and, for F ● dS, F = ρv, a common mistake is to conflate fluid density ρ with the parametric representation of x2 + y2 + z2 in S. Remember that a variable name can represent different expressions in different contexts. Consider renaming instances of a variable name that appears in multiple contexts within the same problem, as is done for radius r and angle θ when represented as parameters of representation r, becoming r(s,t) instead of r(r,θ).


    –


    When computing the surface integral for a vector field against surface S represented parametrically in cylindrical or spherical coordinates, a common mistake is to express dS as rs x rt rdrdt or rθ x rΦ ρ2sin(Φ)dΦdθ. Remember that dS only includes r or ρ2sin(Φ) if S is parameterized and the partial derivatives are expressed in Cartesian coordinates and converted to cylindrical or spherical coordinates by replacing instances of Cartesian variables with their cylindrical or spherical counterparts.


    –

    When computing the surface integral for a vector field against surface S represented parametrically as r(s,t), a common mistake is to forget that the orientation of S determines polarity. Remember that if S points upward above the xy-plane, then the z-component of the cross product rs x rt must be positive. If S points downward below the xy-plane, then the z-component of the cross product rs x rt must be negative. If S points away from the z-axis, then the x- and y- components of the cross product rs x rt must be positive. If S points toward the z-axis, then the x- and y- components of the cross product rs x rt must be negative. If any of the above conditions are false, for the integrand of the surface integral, F ● dr = – (F ● rs x rt) dsdt = F ● – ( rs x rt) dsdt = F ● rt x rs dsdt.


    –


    Applications


    When interpreting the motion of a satellite in circular orbit, a common mistake is to assume that the satellite maintains its speed by forward acceleration. Remember that a satellite with constant speed v has no tangential (forward) acceleration a = dv/dt, where the derivative of constant v is 0. In that case and generally-speaking, the only force acting on a satellite in circular orbit is centripetal (central) acceleration caused by gravitational force. The force being perpendicular to the direction of tangential velocity has neither positive nor negative effect on speed. Despite being in free fall, the reason that the satellite is not pulled to the ground is that its tangential velocity produces a trajectory that matches the curvature of the planet, sufficient to offset the constant gravitational pull toward the planet center which balances the angular motion of the satellite.


    –


    When interpreting the non-motion of a person seated on a chair, a common mistake is to assume that no horizontal forces act on the person, and that spinal compression is generated exclusively by the combined effect of the gravitational force pulling downward toward the plane of Earth and the normal force pushing upward from the plane of the seat. Remember that a person seated in a perfectly smooth-surface chair likely (depending on the angle of the seat and the posture of the person) slips at a speed no greater than v = ∫ a dt where a = Fnet/m as the weight of the person’s legs pull their body forward. When one subject is in contact with and moves across a second stationary subject, particles of the moving subject push against particles of the stationary (non-smooth) subject, while particles of the stationary subject push back against those of the moving subject, in this case, producing the effect of static friction fs parallel to the plane of the seat.

  • [+] Create a synthesizer
    August 18th, 2019

    This walkthrough addresses recording programmatically-generated sounds with the Web Audio and MediaStream Recording APIs, as demonstrated by a simple synthesizer app written with jQuery. For those unfamiliar with jQuery syntax, an interactive tutorial is available at FreeCodeCamp. Walkthroughs from Soledad Penadés and Marc Gauthier are reliable resources for understanding, independently, the sound -recording and -generating processes in greater detail.

    Firefox Users: A bug currently prevents gain ramping. Until the bug is resolved, a Chromium-based browser is recommended.

    Setup

    Create views

    Consider what actions comprise each broader functionality. While generated sounds may be fed directly to the recorder, playing back the sounds as they are being recorded makes sense, as does providing options for generating different sounds and controlling the length of the recording.

    In the context of a synthesizer app, views correspond to keys for playing different musical notes, in addition to the aforementioned controls.

    Create global fields

    The Web Audio API is built around AudioContext, which handles the creation as well as storage of one or a series of a processing module called an AudioNode. Examples of an AudioNode, which is covered later, include sources such as an Oscillator, an intermediate processor such as BiquadFilter and Gain, and a destination which represents the result of all audio in the context. Because the audio in this example has two destinations -- speaker and recorder -- corresponding nodes are created.

     // I/O
     let context = false;
     let speakerNode = false;
     let recorderNode = false;

     // Prefs
     /* e.g. wave type, pitch */

     // View IDs
     /* e.g. play, stop, record, player, display, sounds, etc. */

    Define permission event

    A common browser policy discourages developers from playing sounds prior to obtaining user permission. A developer is free to approach this objective in the manner they see fit, so long as their solution involves user engagement with the page; without which, an error is likely to be thrown. A straightforward implementation might involve listening for a click or key event before attempting to play sounds. 

    $(document).ready(function() {
      if (!context) { 
        $(display).on('click', initializeAudioContext);
        $(document).on('keypress', initializeAudioContext);
      }
    }

    Request AudioContext from browser with Promise

    A Promise provides a convenient interface for associating asynchronous action with dependent logic. A Promise is constructed with an executor that accepts as arguments a handler for, each, resolution and rejection of the action. 

    function getAudioContext() {
       return new Promise((resolve, reject) => { 
           resolve(
             new (window.AudioContext || window.webkitAudioContext)()
           )
           reject(
             "Your browser rejected a request to access the Web Audio API, a required component"
           )
         }
       );
     }

    When the asynchronous action is complete, the Promise is said to be settled -- either fulfilled or rejected. If the Promise is fulfilled, the value resulting from the executor resolve handler is accessible from the .then() method handler argument, where it can be accessed by dependent logic. Likewise, a rejected Promise avails the reason resulting from the executor reject handler by way of the .catch() method which also accepts a handler argument, and these method calls are chained from the root Promise.

    function initializeAudioContext(listener) {
  getAudioContext()
    .then(
      value => {
        context = value;
        speakerNode = context.destination;
        connectAudioDestination();
        ...
    }).catch(
      reason => {
        console.log(reason);
    });
}

    Generate sounds

    Instantiate Oscillator from AudioContext

    An Oscillator produces a constant tone representing a periodic waveform at a certain frequency. A sine waveform at a frequency of 440Hz is the default. The .start() method indicates the time at which the Oscillator should begin playing the tone, which is immediately if no argument is provided. 

    let osc = context.createOscillator();
    osc.connect(speakerNode);
    osc.start();

    Configure Oscillator to produce distinct sounds

    Assigning distinct values to the type and frequency properties changes the tone. Gain affects the volume of the tone, which is instantiated by invoking createGain() from the given AudioContext.

    Gain inherits methods from the AudioParam interface that gradually increase or decrease the property value -- in this case, volume -- over time. Each method accepts two arguments: the targeted value, and the number of seconds from the previous processing event to reach the target. Invoking one of these methods from gain can be useful for fading out volume.

    As, in nature and generally-speaking, sound intensity decreases at an exponential rate between two points, an exponential rather than a linear ramp is more suitable.

    function frequencyToSound(frequency) {
  ...
  osc.connect(speakerNode);
  let  gain = context.createGain();
  gain.gain.exponentialRampToValueAtTime(
    0.00001, context.currentTime + 3.5
  );
  gain.connect(speakerNode);
  osc.connect(gain);
  osc.type = 'sine'; // can also be 'square', 'triangle', 'sawtooth', or 'custom'
  osc.frequency.value = frequency; // can be between -22050 and 22050 inclusive
  ...
}

    For greater control, intermediate processing can be augmented by a BiquadFilterNode.

Other properties including waveform type and pitch (as a frequency multiplier) could be extrapolated by the same means.

    $(sound).on('click', () => {
  let id = '#' + event.target.id;
  ...
  playPad(id);

$(document).keypress(e => {
  let key = e.which;

})

  let sound = frequencyToSound(frequency);
  sound.start(0);
}

  switch(id) {
    case viewId: return 1047;

    Record sounds

    Instantiate MediaRecorder from MediaStream

    A MediaRecorder captures an audio or video MediaStream and saves it as a Blob of media data that can be decoded by a media player. Calls to the start(), pause(), resume() and stop() methods control the length of the recording, with corresponding handlers fired at each call. A MediaStream can be generated as a destination from a sensory device or a compatible source node.

    function captureMediaStream(stream) {
      let recorder = new MediaRecorder(stream); 
      recorder.start();
      ...
    }

    The getUserMedia() method attempts to gain access to a MediaStream from the sensory devices detected from the browser, which by default should generate a prompt requesting user permission. The return value is a Promise which, if fulfilled, resolves the stream to any dependent logic from the then() method handler argument.

    function getUserMedia() {
  return navigator.mediaDevices.getUserMedia({ audio: true, video: true });
}

getUserMedia().then(stream => captureMediaStream(stream));

    In the next section, the stream will be provided by an output node generated from the AudioContext associated with the sound-generating Oscillator.

    Configure MediaRecorder for playback and download

    Before the recording can be played back or downloaded, the data must be compiled. A data set can be built from ondataavailable and finalized in onstop.

    The ondataavailable handler passes a data-storing event that spans the time since, either, the previous handler callback, or, the recording began. The callback occurs either when the recording ends, as well as during the recording if, either, start() is passed a timeslice interval argument or requestData() is invoked. Storing the event data in an array makes sense where multiple data chunks may be returned during a single recording.

    The onstop handler accesses the data array to instantiate a data Blob that is accessed from a URL created with URL.createObjectURL(). The URL can then be assigned to the source attribute of an audio player element and hyperlink reference attribute of an anchor download element.

    let chunks = [];

recorder.ondataavailable = e => chunks.push(e.data));

recorder.onstop = e => {
  let blob = new Blob(
    chunks, { 'type' : 'audio/ogg; codecs=opus' }
  );
  chunks = [];
  let url = URL.createObjectURL(blob);
  $(player).attr('src', url);
  $(download).attr('href', url);
  ...
};

    Record generated sounds

    A few minor modifications to the process for creating the sound generator and recorder establishes a connection between the two elements. The difference stems from adding an extra destination node.

    The AudioContext destination property is assigned an AudioDestinationNode, while the getMediaStreamDestination() method returns a MediaStreamAudioDestinationNode. The AudioDestinationNode outputs directly to the speaker, while the MediaStreamAudioDestinationNode contains a stream properly that can be fed to the recorder.

  • [+] Test a servlet
    July 26th, 2016

    This implementation of a JUnit test for servlet doPost() method relies only on the Mockito library for mocking up instances of HttpRequest, HttpResponse, HttpSession, ServletResponse and RequestDispatcher.

    Replace parameter keys and JavaBean instance with those that correspond to values referenced in the associated JSP file from which doPost() is called.

    Mockito Maven dependency: 

    <dependency>
      <groupid>org.mockito</groupid>
      <artifactid>mockito-all</artifactid>
      <version>1.9.5</version>
</dependency>

    JUnit Test

    import javax.servlet.RequestDispatcher;
import javax.servlet.ServletContext;
import javax.servlet.ServletException;
import javax.servlet.http.HttpServlet;
import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import javax.servlet.http.HttpSession;

import java.io.IOException;

import static org.junit.Assert.assertFalse;
import static org.junit.Assert.assertTrue;
import static org.mockito.Mockito.*;

/**
 * Unit tests for the {@code StockSearchServlet} class.
 * @author bob@coded.art
 */
public class StockSearchServletTest extends HttpServlet {
    // private fields of this class
    private static HttpServletRequest request;
    private static HttpServletResponse response;
    private static StockSearchServlet servlet;
    private static final String SYMBOL_PARAMETER_KEY = "symbol";
    private static final String STARTRANGE_PARAMETER_KEY = "startRange";
    private static final String ENDRANGE_PARAMETER_KEY = "endRange";
    private static final String INTERVAL_PARAMETER_KEY = "interval";
    private static final String SERVICETYPE_PARAMETER_KEY = "serviceType";

    /**
     * Sets up the logic common to each test in this class
     */
    @Before
    public final void setUp() {
        request = mock(HttpServletRequest.class);
        response = mock(HttpServletResponse.class);

        when(request.getParameter("symbol"))
                .thenReturn("AAPL");

        when(request.getParameter("startRange"))
                .thenReturn("2016-04-23 00:00:00");

        when(request.getParameter("endRange"))
                .thenReturn("2016-07-23 00:00:00");

        when(request.getParameter("interval"))
                .thenReturn("DAY");

        when(request.getParameter("serviceType"))
                .thenReturn("WEB");

        String symbol = request.getParameter(SYMBOL_PARAMETER_KEY);
        String startRange = request.getParameter(STARTRANGE_PARAMETER_KEY);
        String endRange = request.getParameter(ENDRANGE_PARAMETER_KEY);
        String interval = request.getParameter(INTERVAL_PARAMETER_KEY);
        String serviceType = request.getParameter(SERVICETYPE_PARAMETER_KEY);

        HttpSession session = mock(HttpSession.class);
        when(request.getSession()).thenReturn(session);
        final ServletContext servletContext = mock(ServletContext.class);
        RequestDispatcher dispatcher = mock(RequestDispatcher.class);
        when(servletContext
		.getRequestDispatcher("/stocksearchResults.jsp"))
		.thenReturn(dispatcher);
        servlet = new StockSearchServlet() {
            public ServletContext getServletContext() {
                return servletContext; // return the mock
            }
        };

        StockSearchBean search = 
		new StockSearchBean(symbol, startRange, endRange, interval);
        try {
            switch (serviceType) {
                case ("BASIC"):
                    search.processData(ServiceType.BASIC);
                    break;
                case ("DATABASE"):
                    search.processData(ServiceType.DATABASE);
                    break;
                case ("WEB"):
                    search.processData(ServiceType.WEB);
                    break;
                default:
                    search.processData(ServiceType.WEB);
            }
        } catch (StockServiceException e) {
            throw new RuntimeException(e.getMessage());
        }
        session.setAttribute("search", search);
    }

    /**
     * Verifies that the doPost method throws an exception 
     * when passed null arguments
     * @throws ServletException
     * @throws IOException
     */
    @Test(expected = NullPointerException.class)
    public final void testDoPostPositive() throws ServletException, IOException {
        servlet.doPost(null, null);
    }

    /**
     * Verifies that the doPost method runs without exception
     * @throws ServletException
     * @throws IOException
     */
    @Test
    public final void testDoPostNegative() throws ServletException, IOException {
        boolean throwsException = false;
        try {
            servlet.doPost(request, response);
        } catch (Exception e) {
            throwsException = true;
        }
        assertFalse("doPost throws an exception", throwsException);
    }
}
